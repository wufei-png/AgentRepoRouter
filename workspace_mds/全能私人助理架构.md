可以把它理解成这样：
OpenClaw 更像“AI 操作台 / 统一入口 / 路由层”，
Temporal 更像“耐久执行引擎 / 后台编排底座”。
它们不是完全同角色，更适合上下配合，而不是二选一。
Temporal 的核心原理很简单：你把一个长任务写成 Workflow，把真正调用外部系统、跑 CLI agent、发请求、读写状态这些步骤写成 Activity。Temporal 会把 Workflow 的执行历史持久化下来，Worker 负责从 Task Queue 里拉任务执行。这样即使机器重启、进程挂掉、网络闪断，Workflow 也能从上次进度继续，而不是整个任务重来。官方文档把这件事叫 Durable Execution，并强调 Workflow 可以持续运行很多年；Worker 则是轮询 Task Queue 执行代码的进程。Temporal Docs+3Temporal Docs+3Temporal Docs+3
这套机制为什么特别适合你说的场景：
你的任务不是“问一句答一句”，而是“5 小时刷新一次订阅”“全仓 code review”“长时间脑暴”“方案迭代优化”“MVP 自动实现”。这些任务有几个共性：会跑很久、会调用很多外部 agent、经常要暂停/继续、还需要失败重试和中途插话。Temporal 正好把这几类问题当主战场：它有持久化的 Timer，可以低成本等很久；有 Schedules 来周期性启动 Workflow，而且官方现在明确更推荐用 Schedules 而不是 Cron Jobs；还支持 Signals / Queries / Updates，也就是运行中的 Workflow 可以被外部继续投递指令、查询状态、动态更新。Temporal Docs+4Temporal Docs+4Temporal Docs+4
所以在你的框架里，Temporal 的位置我建议是：
用户入口层：OpenClaw
耐久编排层：Temporal
实际执行层：Claude Code / Codex / Cursor / OpenCode / 你自己的 skill runner / MCP tools
知识和工具层：MCP、本地索引、向量库、Git、Issue、Docs
也就是说，Temporal 不取代 OpenClaw 的 UI、会话入口、工作区路由能力；它更像 OpenClaw 背后的“任务操作系统”。

Temporal 的简要架构
Temporal 你可以先只记 5 个概念：
1. Workflow
就是“任务蓝图 + 当前任务实例”。
比如一个全仓 review Workflow 可以写成：


扫描仓库


按模块切分


并发派给多个 agent


收集结果


去重


综合


生成总结


如果你追加要求，再继续跑第二轮


Workflow 的关键不是“能编排”，而是它的状态是耐久的。Temporal Docs+1
2. Activity
就是真正有副作用的步骤。
比如：


调用 Claude Code CLI


调用 Codex CLI


执行 git diff


写文件


调用 embeddings / reranker


发 webhook 给 OpenClaw


这些都应该是 Activity，不该直接塞进 Workflow 主逻辑。Temporal Docs+1
3. Task Queue
不同类型的任务进不同队列，不同 Worker 去消费。
你甚至可以按 agent 类型分队列：


review-claude


review-codex


ideation-gemini


workspace-python


kb-rag


这样你就得到了很清晰的 agent 集群路由层。Temporal Docs
4. Worker
就是实际干活的进程。
一个 Worker 可以专门负责：


某个 repo


某种 agent


某类任务


某个 GPU/CPU 资源池


Temporal 不替你当 agent，它管理的是“谁什么时候干什么、失败怎么办、如何恢复”。Temporal Docs
5. Signals / Queries / Updates / Timers / Schedules
这是做“活系统”的关键：


Signal：运行中给任务发新指令
例如“别扫整个仓库了，只扫 payments 模块”


Query：问任务现在做到哪了


Update：更强语义的运行时修改


Timer：耐久等待


Schedule：每 5 小时触发一次，而不是靠系统 cron 硬顶


这几个能力非常适合你做“长期运行 agent 集群”。Temporal Docs+3Temporal Docs+3Temporal Docs+3

- 不同 CLI 启动时注入对应 profile

这样：


- 共享服务只维护一次定义
- 每个 CLI 仍然用自己原生 MCP 能力
- 不强依赖某个 gateway 产品
- 将来替换 OpenCode / Claude Code 也容易


## 所以 mcpport 适不适合你


### 适合拿来收口的


把这些放到 mcpport 之类网关后面比较合理：


- GitHub MCP
- Sentry MCP
- 文档检索 MCP
- 团队 API 聚合 MCP
- 远程数据库读接口
- 需要统一 OAuth / 统一审计的服务

因为 mcpport 的定位本来就是 **gateway & registry**，偏向远程暴露、集中访问和连接边缘服务。 [Model Context Protocol](https://modelcontextprotocol.io/specification/2025-06-18/architecture?utm_source=chatgpt.com)


### 不适合强行收口的


这些不建议全放 mcpport 后面：


- 本地 filesystem MCP
- 本地 shell MCP
- 本地 repo 操作 MCP
- workspace 私有 skill / tool
- 和当前 cwd 强绑定的工具

这些更适合就地 STDIO。



## 结合你现在的多 agent 架构，我建议这样落


### 方案：双层 MCP


**共享 MCP 层**
一个统一 gateway，给所有 agent 复用：


- github
- docs/context7
- sentry
- knowledge-search
- issue-tracker

**本地 MCP 层**
每个 agent / workspace 自己保留：


- filesystem
- shell
- git-local
- repo-index
- browser
- local-db

然后由你的 manager 做**工具裁剪**：


- `Claude Code`：更偏审查/分析，就挂 `github + docs + local repo`
- `OpenCode`：更偏实现，就挂 `filesystem + shell + local repo + docs`
- `Codex`：更偏代码生成/修改，就挂 `filesystem + git-local + test runner`
- `Research agent`：只挂 `docs + kb-search + browser`

这样就不会出现“4 个 agent 每个都重复吃一整套 MCP 上下文”。



## 一个实用判断标准


### 用统一网关


当某个 MCP server 满足这几个条件时：


- 多个 agent 都会用
- 不依赖当前本地 cwd
- 需要统一认证
- 适合远程服务化
- 希望做审计/限流/共享缓存

### 保持本地直连


当某个 MCP server：


- 强依赖本地文件或 workspace
- 只给一个 agent 用
- 对延迟敏感
- 带高权限本地操作
- 不适合跨机器暴露


## 结论


你这个问题的正确答案不是：


**“每个 agent 都配一套”**
也不是
**“所有 MCP 全都统一进 mcpport”**


而是：


**共享型 MCP 统一收口，本地型 MCP 就地保留；再用 profile / 配置生成减少重复。**


一句话版：


**把 mcpport 当“共享服务入口”，不要当“所有 MCP 的唯一运行时”。**
